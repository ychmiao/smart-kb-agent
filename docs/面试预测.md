# 面试预测：Smart KB Agent 项目

> 基于 RAG 的智能知识库问答系统，涵盖文档解析、语义分块、向量检索、多轮对话、多模型熔断降级和 SSE 流式输出。

---

## 一、项目核心面试题（一问一答）

### Q1：请介绍一下你这个项目的整体架构？

**答：**

这个项目是一个基于 RAG 的智能知识库问答系统，整体采用 **Spring Boot 单体模块化架构**，按领域划分为 user、kb、document、llm、search、chat 六个核心模块。

**文档入库链路**：用户上传 PDF/DOCX/MD/TXT 文件 → 存入 MinIO 对象存储 → MySQL 记录处理状态 → 异步线程池触发 Tika 文本解析 → SemanticChunker 按 500 字 + 100 字重叠做语义分块 → Qwen Embedding 向量化 → Milvus（HNSW + COSINE）入库。

**问答链路**：用户提问 → QueryRewriteService 用 LLM 做意图识别和查询重写（判断是否需要检索 + 指代消解）→ RetrievalService 做 Embedding + Milvus 向量检索 Top5 → 构造 RAG Prompt → LlmGatewayService 流式调 DeepSeek（失败熔断降级到 Qwen）→ SSE 推送 rewrite → token* → sources → done 事件流。

**技术栈**：Java 17 + Spring Boot 3.2 + MyBatis-Plus + MySQL 8 + Redis 7 + Milvus 2.4 + MinIO + Resilience4j + WebFlux SSE。所有基础设施通过 Docker Compose 一键启动。

**设计原则**：单体不微服务、无状态 JWT 认证、所有 LLM 调用经过统一网关层、异步非阻塞处理文档和持久化。

### Q2：为什么选择 RAG 而不是直接让大模型回答？RAG 有哪些优缺点？

**答：**

选择 RAG 的核心原因有三个：

1. **知识私域化**：企业/个人的私有文档（技术手册、产品文档、研究报告）不在大模型训练数据中，直接问模型不知道。RAG 通过检索将私域知识注入 Prompt，让模型基于给定材料回答。
2. **降低幻觉**：模型可能编造不存在的知识。RAG 将回答约束在检索到的文档片段范围内，Prompt 明确要求"找不到相关信息时如实告知"，大幅降低幻觉。
3. **可更新、可引用**：文档更新后重新分块入库即可，不需重新训练模型。回答末尾标明【参考来源：文档名】，用户可以溯源验证。

**缺点与应对**：
- **检索质量决定回答质量**：如果 Embedding 不准确或分块策略不合理，召回的内容不相关，回答必然差。应对：使用 Qwen Embedding + Milvus HNSW 索引 + 语义边界分块。
- **上下文窗口占用**：检索到的文档片段占用 Prompt 的 Token 长度。应对：控制 TopK=5，分块大小 500 字。
- **端到端延迟增加**：多了 Embedding + 向量检索两步。应对：异步非阻塞处理 + SSE 流式输出"首 token 优先"。

### Q3：聊一下 LLM 网关的多模型熔断降级设计？为什么需要同时用 DeepSeek 和 Qwen？

**答：**

LLM 网关层（`LlmGatewayService`）是业务代码调用大模型的**唯一入口**，不允许跳过它直接调客户端。

**双层降级策略**：

1. **同步 Chat 降级**：优先 DeepSeek，若抛出 `LlmProviderException`（网络超时、HTTP 4xx/5xx、熔断器开路），则降级到 Qwen。若两者都失败，抛出 `AllLlmProviderFailedException`，最终给用户返回"AI 服务暂时不可用，请稍后再试"。

2. **流式 Chat 降级（关键设计）**：流式降级额外需要考虑"已经输出的 Token 能不能撤回"。方案是维护请求级 `AtomicBoolean hasEmittedAnyToken`：
   - 首 Token 前失败 → 无感知降级到 Qwen
   - 首 Token 后失败 → 禁止切换 Provider，发 SSE error 事件提示用户重新提问

   这避免了用户看到前一段是 DeepSeek 回答风格、后一段突然变成 Qwen 风格这种割裂体验。

**熔断器**：基于 Resilience4j CircuitBreaker，DeepSeek 和 Qwen 各一个独立实例。配置为 50% 失败率阈值、10 次调用窗口、60 秒开路时间、3 次半开探活。熔断快速跳过不健康供应商，避免无效重试拖慢响应。

**为什么同时用两家**？
- DeepSeek 是主模型，能力强、价格低
- Qwen 是备用模型，同时承担 Embedding 职责（DeepSeek 不提供 Embedding 接口）
- 双层供应商避免单点故障——如果 DeepSeek 服务中断，系统仍可用（降级到 Qwen）

### Q4：流式输出场景下为什么不能无脑做多模型降级重试？你们怎么解决的？

**答：**

这是一个很容易忽略但实际很严重的问题。

**不能无脑降级的原因**：

如果 DeepSeek 已经通过 SSE 吐出了部分 Token 给前端，此时切换到 Qwen 重新生成整段回答，用户会看到：
- **重复内容**：DeepSeek 已说了"根据您提供的文档"，Qwen 开头又说"根据您提供的文档"
- **语气风格突变**：DeepSeek 偏简洁，Qwen 偏详细，拼接在一起很不自然
- **结论不一致**：DeepSeek 说了"方案 A 可行"，Qwen 重新分析后说"方案 A 有风险"

**解决方案**：

在 `streamChat()` 方法内部维护一个请求级局部变量 `AtomicBoolean hasEmittedAnyToken`：
- 初始值为 `false`
- 当 Flux 发出第一个 Token（`doOnNext`）时，通过 `getAndSet(true)` 原子置为 `true`
- 在 `onErrorResume` 降级判断点：

```java
if (hasEmittedAnyToken.get()) {
    // 已有内容输出 → 禁止切换 Provider
    return Flux.error(new LlmStreamInterruptedException());
} else {
    // 尚无内容输出 → 无感知降级到 Qwen
    return streamChatProvider(QWEN, ...);
}
```

**为什么用 `AtomicBoolean` 而不是 `volatile boolean`？** Reactor 的 `doOnNext` 和 `onErrorResume` 可能运行在不同的线程上，需要保证可见性和原子性。

### Q5：查询重写的作用是什么？你们是怎么实现的？

**答：**

**作用**：用户在多轮对话中的问题往往依赖上下文（指代消解），比如先问"什么是 RAG？"再问"它的优点是什么？"——第二问的"它"指代 RAG。如果把"它的优点是什么"直接拿去检索，召回效果会很差。查询重写就是把这种**依赖上下文的指代问题，转成独立的自包含检索语句**。

同时还要判断**当前问题是否需要检索**——打招呼、闲聊、简单计算等不需要查知识库。

**实现**：

在 `QueryRewriteServiceImpl` 中，取最近两轮对话（4 条消息）作为上下文，构造一个 Prompt 让 LLM 输出严格 JSON：

```json
{
  "needRetrieval": true,
  "rewrittenQuery": "RAG 技术的优点"
}
```

决策规则：
- 事实性、专业性问题 → `needRetrieval: true`
- 打招呼、感谢、闲聊、简单数学/创意写作 → `needRetrieval: false`
- 不确定时 → 默认 `true`（宁可多检索，不让模型编造）

**容错**：如果 LLM 返回的 JSON 解析失败或格式异常，降级策略是 `needRetrieval=true` + 保留原始问题。这确保了即使重写失败，系统至少能按原始问题去检索，不至于答非所问。

### Q6：语义分块策略是怎么设计的？为什么 chunk 大小选了 500 字？

**答：**

分块质量直接影响 Embedding 的语义匹配精度和检索召回率。

**策略（优先级递减）**：

1. **按段落分割**：优先找连续两个换行 `\n\n`，天然语义边界
2. **按句末标点分割**：段落过长时，倒序找 `。！？.!?` 等句末边界
3. **硬截断**：以上都没找到时，在 500 字符处硬切

**为什么 500 字？**

这是三个因素之间的平衡：
- **太小（<100 字）**：语义不完整，单个 chunk 承载的信息量不足，难以匹配用户提问的语义
- **太大（>1000 字）**：可能混合多个主题，语义向量被平均化；同时占用过多 Prompt 上下文窗口
- **500 字**：大约是中文 3-5 段、一个完整知识点的量，语义相对完整。配合 100 字重叠，确保跨边界的信息不会丢失

**重叠窗口的作用**：防止关键信息正好落在两个 chunk 的切割处被切散。100 字重叠意味着相邻 chunk 共享约 20% 的内容，任何信息至少完整出现在一个 chunk 中。

### Q7：为什么用 SSE 而不是 WebSocket？

**答：**

SSE（Server-Sent Events）和 WebSocket 都是服务端推送技术，选择 SSE 的原因：

1. **单向性匹配**：本项目主要是服务端把 LLM Token 流单向推送给前端，没有前端频繁向服务端发送消息的需求。SSE 天然适合服务端→客户端的单向增量输出。
2. **协议简单**：SSE 基于标准 HTTP，不需要握手升级协议。前端用 `EventSource` API 即可接收，不需额外库。WebSocket 需要 `ws://` 协议、STOMP 子协议等，复杂度更高。
3. **断线重连原生支持**：SSE 规范自带 `last-event-id` 和自动重连机制。WebSocket 需要自己实现心跳和重连。
4. **兼容性**：SSE 在 HTTP/1.1 和 HTTP/2 下都工作良好。

**事件流设计**：
```
rewrite → token* → sources → done
```
- `rewrite`：查询重写结果（是否需要检索 + 改写后的问题）
- `token`：LLM 的回答片段（零个或多个）
- `sources`：引用来源列表（docId + 文件名 + 摘要）
- `done`：流结束标记

每个事件是 JSON 格式，客户端按 `type` 字段区分事件类型。这种设计将回答过程阶段化，前端可以在收到第一个 token 就开始渲染，无需等待完整回答。

### Q8：你们的异步文档处理是怎么设计的？为什么用三个线程池？

**答：**

**异步文档处理流程**：文件上传接口（`POST /api/document/upload`）只做校验 + MinIO 存储 + MySQL 记录创建，然后立即返回文档 ID。真正的 Tika 解析、语义分块、Embedding、Milvus 入库走 `@Async("documentTaskExecutor")` 异步执行，不阻塞上传接口。

这种"同步接口 + 异步处理 + 状态轮询"的模式，可以秒级返回上传结果，用户体验好。

**为什么三个线程池？**

核心原因是**隔离性**——不同类型任务相互不应影响：

| 线程池 | 核心/最大 | 队列 | 用途 |
|--------|----------|------|------|
| `documentTaskExecutor` | 2 / 4 | 100 | Tika 解析、分块、Embedding、Milvus 入库 |
| `llmLogTaskExecutor` | 1 / 2 | 1000 | LLM 调用日志异步写入 |
| `chatPersistenceTaskExecutor` | 1 / 2 | 500 | SSE 流结束后持久化问答对 |

为什么要隔离？
- **文档处理**耗时长（可能数分钟），如果和日志共用线程池，大量文档处理会阻塞日志写入
- **日志写入**请求量大但单次快，配单独池保证不被其他任务饿死
- **问答持久化**需要尽快完成（用户可能在流结束后立即刷新页面），不能等文档处理的慢任务

三个池都设置 `awaitTermination(30, TimeUnit.SECONDS)` + `RejectedExecutionHandler.AbortPolicy`，优雅关闭 + 快速失败。

### Q9：JWT 无状态认证是怎么设计的？和 Session 相比有什么优劣？

**答：**

**方案**：双 Token 机制
- **Access Token**（15 分钟有效期）：存储在 JWT payload 中的 `userId`、`username`，每次请求通过拦截器验证签名和有效期
- **Refresh Token**（7 天有效期）：当 Access Token 过期时，通过 `/api/auth/refresh` 接口换取新 Token

**签发流程**：用户登录 → 验证密码（BCrypt）→ 签发 JWT → 返回 `{accessToken, refreshToken}`

**验证流程**：`JwtAuthenticationInterceptor` 拦截 `/api/**`（排除注册/登录/刷新路径）→ 从 `Authorization: Bearer xxx` 头提取 Token → 校验签名和过期时间 → 设置 `UserContext`

**为什么不用 Session？**
- **无状态**：JWT 不占用服务端内存，不需要 Redis 存储会话，水平扩展时不需要 Session 同步
- **跨平台**：移动端、浏览器、CLI 都可用 Token 认证

**劣势与应对**：
- **无法主动撤销**：签发后直到过期前都有效。应对方案是 Access Token 只有 15 分钟，过期后必须走 Refresh；如果需要强撤销能力，可以引入 Redis 黑名单。
- **Payload 泄露风险**：JWT 只签名不加密，payload 中的信息是 Base64 编码明文。本项目仅在 payload 中放 userId 和 username，不存放敏感信息。

### Q10：你们是怎么做文档删除的？为什么先删 Milvus 再删 MinIO 最后删 MySQL？

**答：**

删除操作采用**三步走、幂等、失败即终止**的策略：

```
Step 1: 删除 Milvus 向量 → Step 2: 删除 MinIO 文件 → Step 3: MySQL 逻辑删除
```

**删除顺序的考虑**：

这个顺序的设计原则是**切最贵的资源先释放**：
- Milvus 存储的是向量数据，是最稀缺、最昂贵的计算资源（每个 kb 一个 collection）。先释放向量数据。
- MinIO 存的是原始文件，存储成本次之。
- MySQL 是逻辑删除（`is_deleted = 1`），代价最小，放在最后。

**幂等**：每一单独执行多次不会出错。Milvus 删除用表达式 `doc_id == ${id}`，如无匹配行则无操作。MinIO 删除不存在的文件不会报错。

**失败即终止**：任一步抛出异常都中断后续步骤，确保不会出现"MinIO 文件已删但 MySQL 记录还在"导致用户看到一条无法下载的记录。

**正在处理中不可删除**：如果文档状态为 `PROCESSING`，抛出 `BusinessException(40910)` 拒绝删除，防止并发读写冲突。

### Q11：Resilience4j 的参数你们怎么配的？熔断器开路后怎么恢复？

**答：**

配置在 `application.yml` 的 `resilience4j.circuitbreaker` 下：

```yaml
deepseek:
  failure-rate-threshold: 50      # 50% 请求失败则熔断
  sliding-window-type: COUNT_BASED
  sliding-window-size: 10         # 统计最近 10 次调用
  minimum-number-of-calls: 5      # 最少 5 次才开始统计
  wait-duration-in-open-state: 60s  # 开路后等 60 秒再尝试半开
  permitted-number-of-calls-in-half-open-state: 3  # 半开状态允许 3 次探活
```

Qwen 配置相同。

**参数设计的考量**：
- **10 次窗口**：适合 LLM 调用这种中频场景。窗口太大（100 次）会让故障持续过久，窗口太小（3 次）误判率高。
- **50% 阈值**：不是 100% 是因为 LLM 偶尔会有 P99 延迟抖动，加上 50% 容错。如果连续 5 次及以上请求失败（超过半数），说明不是偶发问题，果断熔断。
- **60 秒开路**：LLM 服务故障通常需要分钟级恢复。60 秒是合理的"观望窗口"——太短频繁进出半开，太长影响可用性。
- **3 次半开探活**：3 次中有 2 次成功即可关闭熔断器，兼顾快速恢复和可靠性。

**熔断器状态转换**：`CLOSED`（正常）→ `OPEN`（熔断，请求快速失败）→ `HALF_OPEN`（等待 60s 后自动进入，允许 3 次探活）→ 成功则回到 `CLOSED`，失败则回到 `OPEN`。

### Q12：聊聊项目中的异常处理体系？

**答：**

项目采用**分层异常处理**，分三层：

**1. 业务层抛出 `BusinessException`**：
```java
public class BusinessException extends RuntimeException {
    private final int code;  // 业务错误码：40xxx 参数/权限类，50xxx 服务端类
}
```
业务方法在检测到非法状态时主动抛出，携带明确的错误码和描述。

**2. `GlobalExceptionHandler` 统一兜底**（`@RestControllerAdvice`）：
处理 8 种异常类型，全部返回统一的 `Result<T>`：
- `BusinessException` → 对应错误码和消息
- 参数校验异常 → 400 + 拼接所有校验失败信息
- 文件超限 → 400 + "文件大小不能超过 50MB"
- 兜底 Exception → 500 + "服务器内部错误"（含完整堆栈日志，不暴露给客户端）

**3. SSE 流中的异常**：`ChatServiceImpl` 通过 `onErrorResume` 捕获，转成 `ErrorSseEvent + DoneSseEvent`，保证 SSE 连接正常结束，前端永不会挂起。

**LLM 层异常体系**：
- `LlmProviderException` → 单个 LLM 供应商调用失败
- `AllLlmProviderFailedException` → 所有供应商都失败，返回 "AI 服务暂时不可用，请稍后再试"
- `LlmStreamInterruptedException` → 流式输出中途中断且已有部分内容输出，返回 "AI 服务暂时不可用，请重新提问"

### Q13：项目中哪里用了 Redis？为什么存近期 10 条消息就够了？

**答：**

Redis 在项目中的两个核心用途：

**1. 会话消息历史缓存**：

存储结构是 Redis List，key 为 `chat:history:{conversationId}`，TTL 7 天。每次 SSE 流结束后，追加用户消息和助手回答。

查询时取最近 10 条（`LRANGE key -10 -1`），传给 QueryRewriteService 做上下文感知的重写，也传给 LLM 作为多轮对话历史。

**为什么 10 条？**
- LLM 上下文窗口有限，超过 10 轮的对话历史既占用大量 Token，又引入噪声
- 大部分问题只需要参考最近 1-2 轮对话就能理解指代关系
- 超过 7 天的历史转入 MySQL 归档，Redis 只做热数据缓存

**2. 缓存降级**：如果 Redis 不可用，查 MySQL 的 `kb_message` 表作为补充。MySQL 也最多取 10 条且 `is_deleted = 0`。

### Q14：你们这个项目你觉得最有技术挑战的点是什么？

**答：**

我认为最有挑战的是两个方面：

**第一，LLM 网关的熔断降级 + 流式输出一致性**。

这个问题看似简单（"DeepSeek 挂了就切 Qwen 呗"），但在流式场景下复杂得多。如果 DeepSeek 已经吐了"根据文档，方案 A"，此时切到 Qwen 重新生成，Qwen 可能说"根据文档，方案 A 具有成本优势"——用户看到的就是"根据文档，方案 A根据文档，方案 A 具有成本优势"这种拼接内容。

解决方案是引入请求级 `AtomicBoolean hasEmittedAnyToken` 状态：首 Token 前无感降级，首 Token 后只报错不重试。这个方案需要结合 Reactor 响应式编程的 `doOnNext`、`onErrorResume` 操作符做精确控制，同时保证 `AtomicBoolean` 的线程安全性和可见性。

**第二，Milvus 向量数据库的集成与并发控制**。

Milvus 的 collection 是按知识库动态创建的（`kb_{id}`），多文档同时入库时需要防并发重复创建。方案是用 `ConcurrentHashMap` 作为锁池：每个 collection 名对应一个 `Object` 锁对象，`synchronized` 双重检查锁定。同时需要处理 collection 创建失败的补偿——如果创建时向量写入失败，要自动清理已创建的 collection 和索引，防止产生"幽灵 collection"。

### Q15：如果让你扩展这个项目，你打算加什么功能？

**答：**

我会优先考虑三个方向：

**1. 检索质量提升——引入 Rerank 重排序层**

当前是 Milvus 直接返回 Top5 的 COSINE 相似度结果。但向量相似 ≠ 语义相关。可以在 Milvus 之后加一个 Cross-Encoder 重排序（Cohere / BGE），将 Milvus 返回的 Top20 精排为 Top5，精度会有显著提升。

**2. 长会话压缩**

目前最多保留 10 条历史消息。如果会话很长（30+ 轮），之前的关键信息可能被截断。可以在对话达到一定长度时，用 LLM 自动生成历史摘要并压缩，保留核心上下文。

**3. Testcontainers 集成测试**

当前测试使用 Mock 替代基础设施，不够真实。可以引入 Testcontainers 在 CI 中拉起真实的 MySQL、Redis、Milvus 容器做集成测试，覆盖完整的文档入库 → 检索 → 问答端到端链路。

## 二、相关技术八股文预测

### 2.1 Spring Boot 相关

**Q：`@Async` 注解的原理？自定义线程池怎么配？**

**A**：`@Async` 底层通过 AOP + `AsyncExecutionInterceptor` 实现。方法调用被拦截后提交到 `TaskExecutor` 执行，返回 `Future` 或 `CompletableFuture`。

自定义线程池用 `@Bean("name")` + `ThreadPoolTaskExecutor`，配置核心线程、最大线程、队列容量、拒绝策略和优雅关闭。然后在 `@EnableAsync` 类上使用 `@Async("name")` 指定。

注意两个坑：① `@Async` 方法不能和调用方在同一个类中（AOP 代理限制）；② 默认 `SimpleAsyncTaskExecutor` 每次创建新线程，生产环境必须自定义。

**Q：`@RestControllerAdvice` 的执行原理？多个 Advice 的执行顺序？**

**A**：`@RestControllerAdvice` 通过 `ExceptionHandlerExceptionResolver` 实现。Spring 容器启动时扫描所有 `@ControllerAdvice` 类，注册其中的 `@ExceptionHandler` 方法。

执行顺序：通过 `@Order` 注解控制。优先级从高到低（值越小越优先），匹配时取**最精确的异常类型**——即先找 `BusinessException` 的 Handler，没有再找其父类 `RuntimeException` 的 Handler。

**Q：Spring Boot 自动配置原理？**

**A**：核心是 `@EnableAutoConfiguration` + `spring.factories`（Spring Boot 3 改用 `AutoConfiguration.imports`）。Spring Boot 启动时读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件，加载所有 `@Configuration` 类，再根据 `@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty` 等条件注解按需创建 Bean。

### 2.2 Spring WebFlux / Reactor / SSE

**Q：Flux 和 Mono 的区别？**

**A**：`Mono<T>` 表示 0 到 1 个元素的异步序列，相当于 `CompletableFuture<T>` 的增强版。`Flux<T>` 表示 0 到 N 个元素的异步序列。两者都是 `Publisher<T>` 的实现，支持操作符链式调用。

**Q：Reactor 的操作符 `doOnNext`、`onErrorResume`、`onErrorMap` 的区别？**

**A**：
- `doOnNext`：副作用操作，对每个元素执行回调但不改变元素本身，不改变 Flux 的语义
- `onErrorResume`：错误恢复，将上游错误转换为一个**新的 Flux** 继续订阅
- `onErrorMap`：错误转换，将一种异常映射为另一种异常再向下传播（不恢复，只是换类型）

**Q：SSE 的协议规范？和 WebSocket 对比？**

**A**：SSE 是 HTML5 规范的一部分，`Content-Type: text/event-stream`。每个消息以 `data:` 开头，可选 `event:` 指定事件类型，`id:` 指定事件 ID（用于断线重连的 `Last-Event-ID` 头），消息以两个换行结束。

SSE 适用于服务端→客户端单向流（如日志推送、LLM Token 流），协议简单、自动重连。WebSocket 适合双向频繁通信（如在线协作编辑、游戏），但协议握手复杂、需心跳保活。

### 2.3 MyBatis-Plus

**Q：MyBatis-Plus 的 `@TableLogic` 逻辑删除的原理？**

**A**：在实体字段上标注 `@TableLogic(value = "0", delval = "1")`，MP 会在 `deleteById()` 等删除方法上自动将 DELETE 语句替换为 UPDATE `is_deleted = 1`，并在查询时自动追加 `WHERE is_deleted = 0`。

注意逻辑删除字段一定要建索引，否则随着数据增长查询性能会劣化。

**Q：分页插件的原理？pageSize 过大会有什么问题？**

**A**：`PaginationInnerInterceptor` 通过拦截 `Executor` 的 query 方法，在执行前先 count 查询总数，再改写 SQL 添加 LIMIT 分页。

pageSize 过大的问题：① count 查询耗时增加；② 一次查出几千条数据占用大量内存和网络带宽；③ 前端渲染卡顿。本项目限制 `maxPageSize = 100`，超过即回退到 100。

### 2.4 Resilience4j 熔断

**Q：Resilience4j 的熔断状态机？滑动窗口的两种类型？**

**A**：三种状态：`CLOSED`（正常）→ `OPEN`（熔断）→ `HALF_OPEN`（探活）→ `CLOSED`（恢复）| `OPEN`（继续熔断）。

两种滑动窗口：
- **COUNT_BASED**：固定最近 N 次调用的统计
- **TIME_BASED**：最近 N 秒内的调用统计

当达到 `failure-rate-threshold` 配置的百分比阈值时，状态转为 OPEN。OPEN 状态等待 `wait-duration-in-open-state` 后自动进入 HALF_OPEN，放行 `permitted-number-of-calls-in-half-open-state` 次探活请求。探活成功率达标则恢复 CLOSED。

**Q：熔断和限流有什么区别？**

**A**：熔断（CircuitBreaker）是**响应式保护**——检测到被调用方已经故障后快速失败，避免调用方资源被耗尽（线程池满、连接池满）。限流（RateLimiter）是**预防性保护**——控制请求速率在本系统能处理的范围内，防止自身过载，和被调用方是否健康无关。

项目中两者可以配合使用：RateLimiter 控制峰值流量，CircuitBreaker 在下游故障时快速失败。

### 2.5 向量数据库 / Milvus

**Q：传统数据库用 LIKE 搜索为什么不行？向量数据库的优势在哪？**

**A**：`LIKE '%关键字%'` 只能做**精确子串匹配**，无法理解语义。比如搜"怎么修改密码"可能匹配不到"重置密码的流程"。

向量数据库将文本转为语义向量，用 COSINE 相似度或欧氏距离衡量文本间的语义相关性。即使没有共同关键词，语义相近的内容也能被召回。

**Q：HNSW 索引的原理？为什么选 HNSW 而不是 IVF？**

**A**：HNSW（Hierarchical Navigable Small World）是一种分层可导航小世界图索引。建索引时构建多层图结构，顶层节点稀疏（长距离跳转），底层节点密集（精确搜索）。搜索时从顶层快速定位到目标区域，然后逐层细化。搜索复杂度接近 O(log N)。

选择 HNSW 而非 IVF 的原因：
- 搜索精度更高（IVF 需要调 `nprobe` 参数，HNSW 出参稳定）
- 支持动态插入（IVF 需要重建聚类）
- `efConstruction` 和 `M` 参数可调，在建索引速度和搜索精度之间平衡

代价是内存消耗比 IVF 大一些（需要存储图结构）。

### 2.6 RAG 技术

**Q：标准的 RAG 流程有哪些环节？**

**A**：
1. **文档预处理**：解析（Tika）→ 分块（Chunking）→ Embedding 向量化 → 存入向量库
2. **用户查询处理**：查询重写（多轮指代消解）→ Embedding 向量化 → 向量检索 TopK
3. **Prompt 构造**：检索到的文档片段 + 用户问题 + 系统指令 → 构造 RAG Prompt
4. **生成回答**：LLM 基于 Prompt 生成，回答末尾需引用来源
5. **流式输出**：SSE 分阶段推送

**Q：RAG 和 Fine-Tuning 的区别和应用场景？**

**A**：
- **RAG**：外部知识注入 Prompt，不改变模型参数。适合：高频更新数据（新闻、内部文档、产品手册）、需要引用溯源、零成本更新。
- **Fine-Tuning**：在特定数据上继续训练模型。适合：固定格式输出（代码生成风格、客服固定话术）、需要学习新领域知识但 RAG 检索覆盖不到的场景。

两者也可以结合：Fine-Tuning 让模型学会输出风格和格式，RAG 注入实时知识。

### 2.7 Java 并发

**Q：`AtomicBoolean` 和 `volatile boolean` 的区别？**

**A**：
- `volatile` 保证可见性但不保证原子性（`set()` 和 `get()` 是原子的，但 `read-modify-write` 如 `if (!flag) flag = true` 不是）
- `AtomicBoolean` 基于 CAS（Compare-And-Swap），`getAndSet()`、`compareAndSet()` 等操作是原子性的，同时保持可见性

本项目用 `AtomicBoolean` 的 `getAndSet(true)` 检测首 Token——既需要原子性的"检测+设置"，又需要多线程间的可见性。

**Q：`ConcurrentHashMap` 的锁池模式是什么？**

**A**：用 `ConcurrentHashMap.computeIfAbsent` 创建每个 collection 名的锁对象，再用 `synchronized(lock)` 做双重检查锁定。`computeIfAbsent` 本身是原子的，确保不会有两个线程为同一个 collection 创建两个锁对象。`finally` 块释放锁对象，避免内存泄漏。

### 2.8 JWT / 认证

**Q：JWT 的结构和签名流程？**

**A**：JWT 由三部分组成：`Header.Payload.Signature`。
- Header：`{"alg": "HS256", "typ": "JWT"}`
- Payload：`{"sub": "userId", "exp": 1712345678, "iat": 1712344778}`
- Signature：`HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)`

验证时重新计算签名与 Token 中的签名对比。`jjwt` 库会自动处理过期验证和签名校验。

**Q：双 Token（Access + Refresh）方案的优点？**

**A**：
- Access Token 有效期短（15 分钟），即使泄露攻击窗口小
- Refresh Token 有效期长（7 天），但仅用于换取新 Access Token，不走业务请求
- 不需要在服务端存储会话状态（无状态）
- 如果检测到 Refresh Token 泄露，可以吊销它（刷新时校验版本号），同时不立即影响用户体验（Access Token 还在有效期）

### 2.9 Redis

**Q：Redis List 的 LRANGE 取最近 N 条的复杂度？**

**A**：Redis List 底层是 quicklist（双端链表），`LRANGE key 0 -1` 是 O(N)。但取最近 10 条（`LRANGE key -10 -1`）只需从尾部遍历 10 个节点，O(1) 级别。

**Q：项目中 Redis 和 MySQL 的数据一致性问题？**

**A**：本项目以 MySQL 为主，Redis 为加速缓存。SSE 流结束时先写 MySQL，再更新 Redis。如果 Redis 写失败（返回 500 给用户），MySQL 已有完整数据；下次查询时从 MySQL 加载即可。这是一种 **Cache-Aside** 模式——容忍短暂的不一致（秒级），但不会丢数据。

### 2.10 MinIO / 对象存储

**Q：MinIO 和传统文件系统存储的区别？**

**A**：
- 传统文件系统：单机存储、受磁盘容量限制、备份迁移困难、不提供 S3 兼容 API
- MinIO（对象存储）：分布式、可扩展（PB 级）、S3 兼容 API、多版本控制、生命周期管理、适合存储非结构化数据（文件、图片、视频）

本项目用 MinIO 存储原始文档文件，对象名规则 `{userId}/{kbId}/{uuid}.{ext}` 天然按用户和知识库组织，便于后续管理和清理。

**Q：上传接口的事务补偿怎么做的？**

**A**：上传顺序是：① MinIO 上传 → ② MySQL 插入记录 → ③ 异步处理。如果第 ② 步失败，catch 中调用 `minioFileStorage.removeQuietly(objectName)` 静默清理已上传的 MinIO 文件。这种"先执行后补偿"的模式虽然不能保证 100% 一致性（例如补偿也失败），但极大概率成功——MinIO 删除失败概率远低于网络故障概率。
